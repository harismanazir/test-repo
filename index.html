📋 Request Lifecycle

Lifecycle Steps 
1. HTTP Request arrives at route handler
2. Route Handler validates domain_guid is a UUID
   └─ Invalid → 400 {"success": false, "errorMessage": "..."}
3. Service Function checks parsed data cache (TTLCache)
   ├─ Cache HIT → return cached data
   ├─ Cache MISS → build SQL, check raw result cache
   ├─ Raw MISS → call Enrichment Studio Client
   ├─ Parse rows → store in both caches
   ├─ Apply in-memory filters (dimension, template, status)
   ├─ Aggregate (group by dimension/template)
   └─ Paginate (slice by offset:offset+limit)
4. Return Pydantic response model → JSON

🔗 Domain Rule Discovery

Rules belong to a domain in two ways:

Direct

rule.domainguids contains the domain_guid

Indirect

Rule's base dataset qualified name matches a Table/View/MaterialisedView in the domain

📊 Rule Status Categories

All queries filter with WHERE r.status = 'ACTIVE'. Among active rules:

PASS → passed
FAIL → failed
NULL/other → inactive

SQL Queries

Three core queries power the entire reporting system.

1️⃣ Summary SQL — Used by /rule-status

Returns a single row with aggregate pass/fail/inactive counts.

SQL
-- Shared CTE: find all rule GUIDs for the domain
SELECT
    COUNT(*) AS total_rules,
    SUM(CASE WHEN category = 'passed' THEN 1 ELSE 0 END) AS passed,
    SUM(CASE WHEN category = 'failed' THEN 1 ELSE 0 END) AS failed,
    COUNT(*) - passed - failed AS inactive
FROM domain_rules

2️⃣ Detail SQL — Used by 5 endpoints

Returns one row per active rule with guid, category, dimension, template, and qualified name.

SQL
SELECT guid, category, dqruledimension,
       dqruletemplatename, dqrulebasedatasetqualifiedname
FROM domain_rules
ORDER BY category, guid

3️⃣ Domain Assets SQL — Used by /assets, /connectors

Returns one row per asset (Table, View, MaterialisedView) with a has_rules boolean flag.

SQL
SELECT
    domain_assets.qualifiedname,
    domain_assets.connectorname,
    domain_assets.databasename,
    domain_assets.schemaname,
    domain_assets.asset_type,
    CASE WHEN rule_base_qns.rule_qn IS NOT NULL
         THEN true ELSE false END AS has_rules
FROM domain_assets
LEFT JOIN rule_base_qns
  ON domain_assets.qualifiedname = rule_base_qns.rule_qn

Caching Strategy

Two TTLCache layers prevent redundant MDLH queries. Within 5 minutes of the first query, all endpoints reuse cached data with zero MDLH calls.

Layer 1 — Raw MDLH Response Cache

Caches raw responses keyed by SHA256 hash of the SQL query.

Key: SHA256(sql_query) Value: {columns, rows, row_count} TTL: 300s Max: 256 entries

Layer 2 — Parsed Detail Data Cache

Parsed list of rule dicts from detail query, reused by 5 endpoints.

Key: domain_guid Value: list[dict] TTL: 300s Max: 256 entries

Layer 3 — Domain Assets Cache

Parsed asset dicts reused by /assets and /connectors endpoints.

Key: domain_guid Value: list[dict] TTL: 300s Max: 256 entries

🔄 Cache Flow for Detail Data

Cache Resolution
_fetch_detail_data(domain_guid)
  ├─ Check _detail_cache[domain_guid]  → HIT → return
  ├─ Build SQL
  ├─ Check _cache[SHA256(sql)]         → HIT → parse, store in _detail_cache, return
  ├─ Query Enrichment Studio           → store in _cache
  └─ Parse, store in _detail_cache, return

📡 Error Flow Through Layers

Error Propagation
Enrichment Studio Client
  │  raises EnrichmentStudioError / ConnectionError / QueryError
  ▼
Service Layer
  │  lets exceptions bubble up (no catch)
  ▼
Route Layer (_handle_enrichment_errors)
  ├─ ConnectionError → 503 "MDLH service unavailable"
  ├─ QueryError      → 400 "Query error: {details}"
  └─ Any other       → 500 "an internal error just occurred"

Asset Classification Logic

For /asset-status and status_by=assets views: if any rule for an asset has category = "failed", the entire asset status is "failed". Otherwise, the asset is "passed". Assets with null/empty qualified names are excluded.
